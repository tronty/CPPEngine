Unity at SIGGRAPH

Renaldas Zioma, August 13, 2015 in Events

Hey folks, we’re in the middle of SIGGRAPH 2015 in sunny Los Angeles.  As usual, we’ve been blown away, seeing what everyone in the community has been working on over the last year.  SIGGRAPH has historically been the first place to see the latest advances in graphics and this year is no exception.

If you’re at SIGGRAPH now, hopefully you’ve had a chance to check out some of our talks. Here’s the full list, with links to slides.

  • RealtimeLive, The Blacksmith by Vesselin Efremov (@vesseff).
  • Moving Mobile Graphics (course page). Optimizing PBR by Renaldas Zioma (@__ReJ__).
  • Overview of Next-Generation Graphics APIs. Porting Unity to new Graphics APIs by Aras Pranckevičius (@aras_p).
  • Authoring of Procedural Environments in “The Blacksmith” by Vesselin Efremov (@vesseff).
  • [Birds of Feather] Vulkan, OpenGL, OpenGL ES with Christophe Riccio (@g_truc).

Additionally we had some studio courses, taught by Arturo Núñez(@ArturoNereu):

  • Learning to Create Interactive Experiences With Unity 5.
  • Unity 5, The Blacksmith: Make Your Own Viking Movie.

We’ll be at the exhibition hall booth for the rest of the conference; come by and say hi! You can find us at booth #544, or maybe see a few of us wandering the halls. If you have questions, please stop us—we’d love to chat.

Written by Timoni West, Aras Pranckevičius and Renaldas Zioma.

    1. Aras Pranckevičius

      What we have now is HLSL as source language, but a very messy (and very different) compilation pipeline for each and every platform. Some use actual HLSL compiler, some use hlsl2glslfork, some use hlslcc, etc. What we want is to have a single frontend and a “middle layer” for all platforms.

      Right now it looks like we’ll keep on using HLSL as the source language for quite a while, so that would mean we need a single HLSL -> intermediate layer frontend. Such a thing does not exist right now (which is why we don’t “have it”, just “want it” :)… but one can dream).

      For the “middle layer”, right now it feels like building it upon SPIR-V makes the most sense. It’s a well defined intermediate representation for both graphics & compute, and there’s a lot of projects underway in the whole ecosystem that are building various tools upon it.

      Not sure I understood your other questions TBH.
