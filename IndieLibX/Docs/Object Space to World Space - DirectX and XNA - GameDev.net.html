IMHO you are confusing several terms above...


The Ship class is an abstraction of an entity. It stores data about an instance of that kind of Ship. This includes its position (and orientation) in (typically) world space. It may further include e.g. a bounding volume for speeding up collision detection, and similar stuff.

Further there is a model of the ship for graphical rendering purposes. This model typically consists of a vertex mesh. The vertices are given w.r.t. a co-ordinate frame, typically so that the model's hull is more or less centered around 0 (notice that this is a vectorial zero, i.e. zero in 3 dimensions) and with its front facing into a specific, usually cardinal direction (e.g. the cardinal z axis direction).

Now, if you would use the model "as is" would mean that it appears in this default orientation and position but in the world space. That would not what you want, because you want it be placed as given by the data in the Ship instance. Hence you need to rotate and translate the model to make its world position and orientation match the data in the Ship instance. This is accomplished by rotating the model into the desired orientation and translating the rotated model to the desired position. (Please notice the order of transformations as it is important!)

Now, if you would apply these transformations on the model's vertices and overwriting the vertices, you would a) use an absolute transformation as a relative one (meaning that you would sum up transformations although you wanted to apply it only once), and b) the model will loose its shape over time because of accumulating small numerical inaccuracies. Therefore, what you want to do is to compose the matrix of the transformations to be applied, and supply it as well as the model's vertices to (in this case) D3D, so that D3D can transform the vertices on-the-fly for rendering without overwriting your original vertices.

In summary, you want D3D let transform the vertices each time it renders another frame. But you want to accumulate player input that effects the ship's position and orientation into the data stored in the Ship instance. And you want to compute the transformation matrix from the said data.


The view matrix by itself is independent on the Ship instance. It is rather dependent on where an observer is located in the world (often called the "eye-point"), and where s/he looks at (often called the "line-of-sight"). Together this is often modeled as a virtual camera. Although this may be attached to your Ship instance (used as a 3rd person camera or from the inside like a 1st person camera), I won't say that you need to create a view matrix for your ship but for a camera set-up.

However, you are right in that the view matrix will have an effect on the rendering, of course. This is because what you get to see on the screen is observer centric: Regardless whether an object is placed at (1,0,0) or at (1000,0,0) in the world, it appears just in front on the screen when the observer is at (0,0,0) or at (999,0,0), resp., assuming that s/he looks into the correct direction, of course. This is accomplished by transforming the entire world so that the observer (or camera) is located at 0 and oriented in standard direction. Hence you need to apply the inverse of the transformation that is used to place the observer (or camera) in the world. And exactly that is the view matrix application.


Coming to updates in world/model space: It is a mathematical fact that the matrix product is not commutative, so A * B != B * A in general. So, if you want to continuously add up translation and rotations to yield in the new current placement data of the Ship instance, you need either to handle orientation and position apart from each other (i.e. sum up rotations on an orientation and sum up translation on a position), or else make sure that the the current delta transformation has the desired effect on the (otherwise) composed placement.

The former way means for step i, that
Ri := Ri-1 * ri
Ti := Ti-1 * ti
Mi := Ri * Ti
when using row vectors (as is typical for D3D), where R means the orientation, r the current rotation, T the position, and t the translation. So you compose the model's world matrix M after having applied the current delta transformations.

The latter way means that you do something like
Mi := Mi-1 * ri * ti
but that isn't exactly what you probably want, because if you set in M recursively
Mi = Mi-2 * ri-1 * ti-1 * ri * ti
you'll see that you rotate the already translated model. Hence you would need to undo the former translation (ti-1) by applying its inverse before applying ri (not to forget to later redo the translation) like so
Mi := Mi-2 * ri-1 * ti-1 * ( ti-1-1 * ri * ti-1 ) * ti
where the parentheses are written only for cosmetic purposes; they are not necessary from a mathematical point of view. The consequence is now, if you continue to recursively set in M, then you'll see that you actually compute
Mi = Ri * Ti
like done in the first way shown above. That's the whole secret of this undo/redo transformation stuff.

As haegarr ( who was kind enough to give a thorough explanation ) mentioned. Order of Matrix Multiplication is important.

D3DXMATRIX world, view, projection, objSpace_to_NDCSpace;

objSpace_to_NDCSpace = world * view * projection; // world, then view then projection

Since you are using d3d, I must ask, are you using shaders or the Fixed function pipleline. If shaders, then study hard my friend, because you will need a much better understanding of this stuff if you plan to write shaders. If not, ie you are using device->SetTransform() then you don't want to concatenate the matrices your self.

D3DDEVICE->SetTransform(D3DTS_WORLD, &world);
D3DDEVICE->SetTransform(D3DTS_VIEW, &view);
D3DDEVICE->SetTransform(D3DTS_PROJECTION, &projection);

You just tell your d3ddevice where they are, and what they are for, and it will do the rest. Easier, but less flexible in the long run.

You take an object from one space into another by applying a space-to-space descriptive transformation and re-interpret the transformed co-ordinates with respect to the reference frame of the other space. Well, to understand this, you must be aware that the world space has in fact absolutely no feature that makes it superior to e.g. what is called a model's local space. This is true for the view space, too, or any other one. For example, if you have a model centered around 0, and you don't apply a world or view transformation to it but render it "as is", then you interpret the model's vertices as given with respect to view co-ordinates (assuming that a projection is applied by D3D). If you apply, say, a translation to it and render it thereafter, then you interpret the translated mesh in view co-ordinates. Whether the translation counts to the local-to-world transformation or to the view transformation is just a convention.

The transformation pipeline applied to a mesh is longish, but the following (written in order for row vectors, as used by D3D)
M * V * P * ...
where M denotes the local-to-world space matrix, V the view matrix, and P the projection matrix, is the part one primarily deals with. It can be interpreted as transforming from the local space into the global space (a.k.a. world) by applying M, transforming from global space into view space by applying V, and from view space further by applying the projection matrix P (whereby e.g. perspective projection happens).

A composite of such 3 matrices is probably what appears named as gWVP in your shader script. It is probably composed by using D3D routines that multiply matrices, and then set as a compact matrix into the GPU. (I don't want to discuss the details because I'm not a D3D programmer.) However, when you want to translate a model in a specific space then look where in the transformation pipeline the space is active and insert the translation there. It will usually be just before V, i.e. as last part inside M.

'Believe82', on 19 Apr 2011 - 5:40 PM, said:

Haegarr, the M*V*P logic made sense but it just resulted in translating the entire terrain along with my ship.

Do i need to write a seperate shader for my terrain and ship?

Perhaps, but not for this reason. Instead you have to push another MVP into the shader. P changes usually never. V changes from frame to frame (assuming the camera is in motion) but is the same for all objects in the current frame. But each object has its own M, and it may change from frame to frame (animated objects) or not (keyword "static geometry"). So MVP usually varies from object to object, and hence need to be set as its own value for each object to render.

You have to compose the transformation on the CPU, set it trough the graphics API either in the FFP or the shader, and let the GPU apply the transformation onto the mesh. That's the usual way (of course, special use cases exist where something else happens).
